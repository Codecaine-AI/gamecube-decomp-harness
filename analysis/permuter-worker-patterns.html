The Worker Loop

1. Establish baseline

Use checkdiff, direct compile, m2c, objdump, similar functions, and local source reading to name the remaining mismatch.

2. Ask permuter

Run non-applying search with bounded iterations, usually jobs=4, often saving a replay recipe.

3. Interpret candidate

Read the diff as a source-shape clue: declaration order, helper boundary, lifetime, stack pressure, or expression scheduling.

4. Humanize or reject

Keep only reviewable source. Reject fake helpers, pragmas, volatile tricks, relocation regressions, and broad generated-looking edits.

5. Validate and record

Rerun target checkdiff, neighbor summary, direct compile as needed, and review lint. Record facts and rejected hypotheses.

Near-Exact Random Sweep Pattern

1. Gate the request Spend scarce permuter slots when the worker already has a stable high-99 baseline, a small mismatch, and a named source-shape suspicion.
2. Fan out seeds Run many bounded, independent recipes against the same packet: same function, same patch/base, varied seed, capped workers, saved replay.
3. Cluster the hints Sort by score delta, then look for repeated shapes across candidates: local order, lifetime, helper split, pointer base, or scheduling pressure.
4. Carry back evidence Return top diffs, scores, recipes, and logs. The worker brings back the idea, writes readable source, and validates target plus neighbors.
Useful hitMultiple seeds converge on the same source-shape move, or one candidate makes a narrow high-99 jump that explains the known mismatch.
Useful missFlat no-improvement across random seeds says the worker should stop blind search and sharpen the hypothesis before spending another slot.
RejectExact-ish output that requires pragmas, fake helpers, relocation churn, or broad unreadable edits stays evidence only, not source to land.

Operational Reading

The dominant pattern is not "permuter solves and worker applies." It is "worker asks the permuter for evidence, turns the evidence into a human source hypothesis, then validates." For a capped remote runner, the sweet spot is near-exact work where randomized sweeps can expose tiny compiler-shape clues without tying up local workers.

Remote Runner Implication

A useful remote packet should include function, optional mutate functions, seed or seed range, max iters, timeout, jobs, base commit or patch, baseline score, and whether to save replay. The response should include ranked candidate diffs, score deltas, replay recipes, stdout/stderr, and enough metadata to explain whether it was useful evidence, no improvement, or rejected output.
